# 音乐编年史报告写作指南

## 报告整体结构

报告应该包含以下部分，按顺序编写：

### 1. 音乐内核总述（开篇）
- 用一个生动的比喻概括用户的音乐灵魂
- 点明贯穿始终的审美线索
- 从第一首歌的收藏日切入，揭示灵魂底色
- 用一句反复出现的核心判断统领全文（如"你从来没有在听音乐，你一直在回家"）
- 字数：400-600字

### 2. 编年史纪元（主体，按年份分段）
- 将用户的收藏历史划分为3-6个"纪元"
- 每个纪元以年份+核心特征命名（如"第一纪元：2015-2016 ——饥饿的灵魂"）
- 每个纪元包含：
  - 数据概览（首数、日均收藏量）
  - 核心发现（最爱歌手、新出现的音乐类型）
  - **关键时刻故事化**：挑出有故事感的具体日期（如跨年夜、某个高频收藏日），描写当时的场景和心理
  - 个性化解读（不是数据罗列，而是洞察）
  - **人生阶段对应**：明确写出该纪元对应的年龄和人生阶段
- 每个纪元400-1000字
- 注意纪元之间要有递进感：从"贪婪探索"到"确认身份"到"沉默修复"到"回望新生"到"和解"

### 3. 音乐琴弦（灵魂画像）
- "如果你的灵魂是一把琴"
- 将用户的音乐品味抽象为5-6根"弦"
- 每根弦对应一类音乐和背后的情感需求
- 命名格式：第X根弦：XX弦（如"史诗弦"、"孤独弦"、"真诚弦"、"温柔弦"、"故乡弦"、"生命力弦"）
- 每根弦包含：代表歌手/歌曲 + 功能描述 + 与人生的对应
- 字数：600-900字

### 4. 灵魂疆域（精神地图）
- 一份歌单画出的精神地图——用户的灵魂住在哪里、去过哪里、想去哪里
- 从音乐中提取3-5个"坐标"，例如：
  - **地理坐标**：北方（贰佰的太原、赵雷的城南）、日本（90年代的ZARD/X JAPAN）
  - **时间坐标**：深夜（凌晨0-4点的收藏记录）
  - **空间坐标**：长歌的疆域（超长歌曲的意义）
- 每个坐标都是一段独立的小品文，有场景、有情绪
- 字数：500-800字

### 5. 峥嵘岁月（年龄线总结）
- 按用户年龄段（每2-3年一段）快速回顾人生弧线
- 每段用一句话概括核心特征 + 2-3句支撑
- 最后一行用**粗体**做全局总结（如"你用11年完成了一条完整的弧线：从'我是谁'到'我知道我是谁'再到'我接受我是谁'"）
- 字数：400-600字

### 6. 下一纪元：音乐预言（推荐章节）

根据用户当前年龄和已分析的音乐品味演进轨迹，为其即将进入的人生阶段推荐音乐并给出哲学释义。

**结构**：
- **纪元命名**：为用户的下一个人生阶段起一个诗意的纪元名称（如"第六纪元：2027-2029 —— 允许自己变化的年纪"）
- **阶段描述**：用 2-3 句话描绘这个人生阶段的可能图景（不预测具体事件，而是描绘心境与状态的自然演变）
- **场景化推荐**：推荐 8-12 首歌，但**不是列表式罗列**，而是：
  - 为每首歌构建一个具体的人生场景（如"凌晨两点，孩子终于睡了"、"35岁生日那天独自散步"）
  - 在场景中自然引入推荐歌曲
  - 解释为什么这首歌属于那个场景和那个年纪
  - 用分隔线（---）隔开不同场景
- **哲学释义**：用一段 200-300 字的哲思为这个推荐章节收尾，阐释音乐与时间、成长、自我的关系

**推荐逻辑**：
- 基于用户现有偏好的"自然延伸"（60%）——同类型但尚未收录的优质作品
- 基于其品味演进方向的"温和探索"（30%）——在其已展现的变化趋势上往前推一步
- 基于人生阶段的"灵魂补给"（10%）——这个年龄段普遍需要的情感滋养

**注意**：
- 推荐的歌曲必须是真实存在的作品
- 不要推荐用户歌单中已有的歌
- 推荐语应延续报告的叙事风格，而非"你可能会喜欢"这种推荐算法口吻
- 哲学释义应与整篇报告的灵魂画像相呼应，形成首尾闭环
- 场景描写要具体、有画面感、有温度

**字数**：800-1500字

### 7. 尾声
- 从第一首歌到最近一首歌、再到"未来的第一首"的呼应
- 对用户做一个完整的"人格素描"——温柔但倔强、独立但柔软、有信仰、一直在成长
- 最后一句留给用户力量和温暖（如"你不需要被治愈。你只需要被看见。"）
- 字数：400-800字

## 写作风格要求

### 语气
- 第二人称叙述（"你"）
- 像一个懂音乐的老朋友在和用户深谈
- 温暖但不煽情，洞察但不说教
- 允许偶尔的幽默和调侃
- 允许偶尔粗口（如引用歌词"操他妈的但我还活着"），保持真实感

### 数据使用
- 数据必须精确（直接引用分析结果中的数字）
- 但数据是为故事服务的，不是为了展示
- 避免"根据数据分析..."这种生硬的表达
- 用具体的场景和细节替代干巴巴的数据罗列
- 关键数字加粗或与场景自然融合（如"871首歌，一个刚发现自己可以拥有整个世界的人"）

### 故事化技巧
- 挑出"高频收藏日"（单日收藏>10首），猜测当时的场景
- 关注跨年夜、特殊日期的收藏
- 关注凌晨收藏——深夜的选择往往更诚实
- 关注重复收藏——同一首歌收两次意味着什么（跨越的年份越长越有意义）
- 关注"突然出现"的歌手——大量集中收藏某个歌手是什么信号
- 关注歌名本身的文学性（如十指流玉的歌名就是情话）
- 关注音乐类型的"断裂"——某年突然从后摇切到民谣说明什么

### 结合用户年龄
- 根据用户提供的年龄，推算每个年份对应的人生阶段
- 将音乐选择与人生阶段结合解读
  - 大学时期：探索、贪婪、什么都想尝试
  - 初入职场：需要力量型音乐（Beyond、五月天）
  - 中期：从探索转向确认，速度放缓，变成"稳定剂"
  - 成熟期：回望、和解、接受，开始关注"下一代"
- 每个纪元开头标注年龄和首数

### 句式特征
- 善用短句制造节奏感："但恰恰就是同一个人。"
- 善用排比制造力量感："Nightwish的北欧神话是庇护所、X JAPAN的18分钟钢琴独奏是庇护所、惘闻的后摇是庇护所。"
- 善用对比制造张力："后摇是你的教堂，贰佰是你从教堂出来之后喝的第一口酒。"
- 善用问句引导思考："能听完这首歌的人，内心一定有一片荒原需要被填满。"
- 金句用粗体标注：**你从来没有在"听音乐"，你一直在"回家"。**

### 个性化原则
- **绝对不要使用模板化的表述**
- 每一段解读必须基于该用户独特的数据
- 如果用户补充了个人信息，要有机地融入分析
- 关注歌单中的"异常值"——它们往往是最有故事的部分
- 对每个歌手/歌曲的解读要具体到"为什么是这首歌/这个人"，而非泛泛而谈

## 报告长度

- 完整报告：4000-8000字
- 如果用户歌单较小（<500首），适当缩短至2500-4000字
- 如果歌单巨大（>3000首），可以适当扩展，但不超过10000字

## 关键数据解读维度

基于 analysis_data.json，重点关注：

1. **year_counts** → 年度收藏量的起伏（哪年最多？哪年断崖？为什么？）
2. **top_artists** → 最爱歌手（前20名必须逐一分析含义）
3. **year_top_artists** → 各年份偏好变化（品味的演进路径）
4. **lang_distribution** → 语言分布（什么比例的日/英/中/韩？说明什么？）
5. **duration_stats** → 时长偏好（喜欢长歌的人和喜欢短歌的人完全不同）
6. **peak_days** → 高频收藏日（那天发生了什么？）
7. **late_night_count** → 深夜收藏量（深夜是另一个人格）
8. **earliest_songs / latest_songs** → 首尾对比（成长轨迹）
9. **decade_distribution** → 听的音乐来自哪个年代（是否怀旧型？）
10. **重复收藏** → 同一首歌出现多次意味着灵魂回声

## HTML 报告设计规范

当需要手工编写精美 HTML 报告时（超越 Markdown→HTML 自动转换），遵循以下设计语言：

### 页面结构
- 每个章节独占一页（`page-break-after: always`）
- 每页有独立的背景色/渐变
- 每页有 2-3 个轻量装饰元素（CSS伪元素实现）
- 每页底部有页码

### 颜色主题（按章节）
- 封面：暖橙渐变 + 唱片/音符装饰
- 音乐内核：浅紫渐变 + 声波装饰
- 各纪元：分别用暖金、红橙、嫩绿、琥珀、淡蓝
- 琴弦：浅紫 + 竖线装饰
- 灵魂疆域：米黄 + 网格/指南针
- 峥嵘岁月：暖棕 + 时间线/沙漏
- 尾声：暖金渐变 + 光芒/心形

### 排版组件
- `.quote-block`：引用块（左侧色条 + 浅色背景）
- `.string-item`：琴弦卡片（左侧彩色边框）
- `.territory-card`：疆域卡片（圆角 + 微边框）
- `.timeline-item`：时间轴项（左侧圆点 + 浅色背景）

### 字体
- 标题：Noto Serif SC (衬线)
- 正文：Noto Sans SC (无衬线)
- 通过 Google Fonts CDN 引入
